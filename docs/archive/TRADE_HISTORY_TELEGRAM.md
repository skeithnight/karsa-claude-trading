# 📱 Karsa Trading Bot: Telegram Trade History & Pagination Architecture

**Date:** October 2023  
**Scope:** Telegram Bot UI/UX, Inline Keyboard Pagination, Safe Text Formatting.  
**Objective:** Replace cluttered, error-prone text dumps with a clean, interactive, and mobile-friendly paginated trade history interface.

---

## 📑 Executive Summary

Previously, the `/history` command dumped all recent trades into a single message using raw HTML tags (`<code>`, `<b>`). This approach suffered from three critical flaws:

1. **Message Bloat:** Exceeded Telegram's 4096-character limit when trade history grew.
2. **Parsing Crashes:** AI-generated reasoning often contains mathematical symbols (e.g., `price < entry`). Telegram's HTML parser interprets `<` as the start of a tag, causing the message to fail or display broken formatting.
3. **Poor Mobile UX:** Dense blocks of text are difficult to read on mobile devices.

**The Solution:** We have transitioned to a **Paginated Inline Keyboard UI** using **Pure Unicode formatting** and **strict text escaping**. This provides a native app-like experience, guarantees zero parsing errors, and keeps the chat history clean.

## 🏗️ 2. Architecture & Data Flow

### Component Interaction

```text
[ User ] ──(/history)──> [ CommandHandler ]
                             │
                             ├─> 1. Fetch Total Stats (Wins/Losses/Net)
                             ├─> 2. Fetch Page 1 Trades (LIMIT 5 OFFSET 0)
                             └─> 3. TradeHistoryFormatter.build_message()
                                     │
                                     └─> [ Telegram API: sendMessage ]
                                             │
[ User ] <──(Interactive Message)─────────────┘
   │
   └─(Clicks "Next")──> [ CallbackQueryHandler ]
                             │
                             ├─> 1. Parse requested page from callback_data
                             ├─> 2. Fetch Page 2 Trades (LIMIT 5 OFFSET 5)
                             └─> 3. TradeHistoryFormatter.build_message()
                                     │
                                     └─> [ Telegram API: editMessageText ]
```

### Key Architectural Decisions

1. **`parse_mode=None` (Pure Text):** We completely disable Telegram's HTML/Markdown parsing for this message. This guarantees that any weird characters generated by the AI (like `<`, `>`, `&`, `_`) will render exactly as typed, preventing 100% of formatting crashes.
2. **Stateless Pagination:** The page number is encoded directly in the button's `callback_data` (e.g., `karsa:history:page:2`). We do not need to store user session state in Redis or memory.
3. **In-Place Editing:** Clicking "Next" uses `editMessageText` instead of sending a new message. This keeps the Telegram chat clean and prevents notification spam.

### 3.2 The Telegram Handlers

Located at `src/bot/handlers/trade_history.py`.

**Command Handler (`/history`):**

* Fetches the global trade statistics (for the summary footer).
* Fetches the first 5 trades from the database.
* Sends the initial message.

**Callback Handler (`karsa:history:*`):**

* Listens for inline button clicks matching the regex `^karsa:history:(page:\d+|noop)$`.
* Extracts the requested page number.
* Calculates the database `OFFSET` (`(page - 1) * 5`).
* Fetches the specific slice of trades.
* **Edits** the existing message in place.

## 🛡️ 4. Security & Robustness

### Preventing Telegram HTML Injection

In the previous iteration, if the AI Judge outputted:
> *"Price is < entry, so we exit."*

Telegram's HTML parser would see `< entry` and throw a `BadRequest: Can't parse entities` error, causing the bot to fail to send the message.

**The Fix:**
By setting `parse_mode=None` and running the AI text through our `escape()` utility, we treat the AI output as **literal plain text**.

* `<` renders as `<`
* `>` renders as `>`
* `_` renders as `_` (instead of triggering italics)

This makes the UI **100% immune to AI hallucination formatting bugs**.
